# UIモック技術選定 解説ドキュメント

> 選定技術スタック: **React 19 + TypeScript + Tailwind v4 + Vite（vite-plugin-singlefile）**
> 対象読者: 非エンジニア・フロントエンド未経験者

---

## まず「なぜUIモックが必要なのか」から

字幕エディタは、**動画を再生しながらリアルタイムで字幕を確認・修正する**ツールです。
これは「ボタンを押したらサーバーが処理して結果を返す」という単純な仕組みでは作れません。

- 動画の再生位置に連動して、該当字幕をハイライトする
- 文字数（CPS）をリアルタイムで計算・表示する
- 字幕テキストを編集したら即座に文字数を再計算する

こういった**「画面上で即座に反応する」処理**を実現するための技術選定が今回の話です。

---

## 技術選定の全体像（料理に例えると）

| 技術 | 料理に例えると | 実際の役割 |
|------|--------------|-----------|
| **React 19** | コンロ・調理器具（メインツール） | 画面の部品を組み立て、状態を管理する |
| **TypeScript** | レシピの分量表記・計量カップ | コードの「型」を定義してミスを防ぐ |
| **Tailwind v4** | 計量スプーン・下ごしらえセット | デザイン（見た目）を効率よく書く |
| **Vite** | 高速オーブン（ビルドツール） | コードをブラウザが読める形に変換する |
| **vite-plugin-singlefile** | 弁当箱（一箱にまとめる） | 全部を1つのHTMLファイルに詰め込む |

---

## 各技術の解説

### 1. React 19 ── 画面の「状態」を管理する仕組み

#### Reactがない世界のイメージ

普通のWebページは「静的な文書」です。HTMLで書いた内容がそのまま表示されます。
動きを付けるにはJavaScriptで「このボタンが押されたらこのテキストを書き換える」と
**一つひとつ手動で命令する**必要があります。

```
// Reactなし（手動で全部やる、つらい）
const text = document.getElementById("subtitle-text")
const counter = document.getElementById("char-count")

text.addEventListener("input", function() {
  counter.textContent = text.value.length + " 文字"  // 手動更新
  if (text.value.length > 40) {
    counter.style.color = "red"  // 手動で色変更
    // 他にも更新が必要な場所があれば全部手動...
  }
})
```

画面の要素が増えるほど「どこを更新したか」の管理が地獄になります。

#### Reactの考え方：「状態が変わったら画面を自動で更新する」

Reactは**「状態（データ）→画面」という一方向の流れ**で動きます。

```
状態: { subtitleText: "Hello world", currentTime: 12.5 }
         ↓ 状態が変わると自動的に画面が更新される
画面: [字幕テキスト表示] [文字数カウンター] [ハイライト位置]
```

開発者は「状態をどう変えるか」だけに集中すればよく、
「どの要素を更新するか」はReactが自動でやってくれます。

#### 字幕エディタでの具体例

```
字幕テキストを編集
  → subtitleText という「状態」が変わる
    → Reactが自動的に以下を更新:
       - テキスト表示エリア
       - CPSカウンター（文字数/表示時間）
       - 制約違反の警告表示
       - すべて自動、コードは「状態を変える」だけ
```

#### React 19 の「19」って何？

Reactはバージョンアップを繰り返しており、2024年末にv19が正式リリースされました。
v19では**Actions**という新機能で非同期処理（データ送信など）が大幅に簡単になっています。
今回のUIモックには直接影響しませんが「最新版を使う」という意味で19を指定しています。

### 2. TypeScript ── コードの「型」を定義してミスを防ぐ

#### 「型」とは何か

型とは「この変数には何が入るか」の宣言です。

```
// TypeScriptなし（JavaScriptのみ）
let cps = "15"   // 文字列なのに数値として計算しようとして壊れる
let cps = 15     // 数値として計算できる

// TypeScriptあり
let cps: number = "15"  // ← ここで即エラーが出る（「数値型に文字列は入れられません」）
let cps: number = 15    // OK
```

#### なぜ字幕エディタで重要か

字幕データは複雑な構造を持ちます：

```typescript
// 字幕セグメントの「型」定義
type SubtitleSegment = {
  id: number
  startTime: number   // 秒数（小数あり）
  endTime: number     // 秒数（小数あり）
  text: string        // 字幕テキスト
  cps: number         // Characters Per Second
  isValid: boolean    // 制約を満たしているか
}
```

型を定義しておくと：
- `segment.starttime`（タイポ）→ 即エラー検出（正しくは `startTime`）
- `segment.text.length / segment.startTime`（時間で割ってしまう）→ 型は合っているが論理ミス
- AIがコードを生成するときも、型定義を参照してミスを抑制できる

#### 「AIによるコード生成のガードレール」とはどういう意味か

LLM（Claude等）がコードを生成するとき、型定義があると：
- 「このプロパティは `number` 型なのに `string` を返していますよ」と気づける
- 型エラーがあればテスト前に発見できる
- 型定義が「仕様書」の役割を果たすので、AIが文脈を正確に理解できる

### 3. Tailwind v4 ── CSSを「クラス名」で書く仕組み

#### 通常のCSS（スタイルシート）との違い

```
<!-- 通常のCSS -->
<div class="subtitle-warning">制約違反</div>

/* 別のCSSファイルに書く */
.subtitle-warning {
  color: red;
  font-weight: bold;
  padding: 4px 8px;
  border-radius: 4px;
}
```

```
<!-- Tailwind CSS -->
<div class="text-red-500 font-bold px-2 py-1 rounded">制約違反</div>
```

Tailwindでは**HTMLのクラス名に直接スタイルを書きます**。
別ファイルを行き来しなくてよいので、小規模プロジェクトでは効率的です。

#### v4 の「v4」って何？

Tailwind v4（2025年リリース）では設定ファイルが大幅に簡略化され、
CSSの書き方も直感的になりました。v3から移行するとプロジェクトの初期設定が楽になります。

### 4. Vite ── 開発を速くする「ビルドツール」

#### 「ビルド」とは何か

ブラウザはTypeScriptを直接読めません。JSXという構文も読めません。
Tailwindのクラス名も、最終的には普通のCSSに変換が必要です。

これらを**「ブラウザが読めるHTML/CSS/JSに変換する作業」がビルド**です。

```
[開発者が書くコード]          [ブラウザが読むコード]
  App.tsx (TypeScript)   →    app.js (JavaScript)
  style.css (Tailwind)   →    output.css (純粋なCSS)
  index.html             →    index.html（完成品）
```

#### Vite が速い理由

従来のビルドツール（Webpack等）はファイルをまるごと処理していました。
Viteは**変更があったファイルだけを即座に更新**する仕組みで、開発中の反映が爆速です。

- Webpack: ファイル保存 → 全体を再ビルド（数秒〜数十秒）
- Vite: ファイル保存 → 変更箇所だけ更新（ほぼ即時）

---

### 5. vite-plugin-singlefile ── 「1ファイル完結」の要

#### これが一番重要な選定理由

通常のViteでビルドすると、このようなファイル群が生成されます：

```
dist/
├── index.html
├── assets/
│   ├── index-Bk3a9F2n.js     ← JavaScriptファイル
│   ├── index-D8kP0l3m.css    ← CSSファイル
│   └── logo-C4v2xW1q.png     ← 画像ファイル
```

これを配布するには**全部のファイルが揃っている必要があります**。
誰かに渡すとき、フォルダごと圧縮して送る必要があります。

#### vite-plugin-singlefile を使うと

```
dist/
└── index.html   ← これ1つだけ！（JS/CSSが全部インライン化される）
```

JSもCSSも画像も、**すべてindex.htmlの中に埋め込まれます**。

#### 「start.bat だけで配布できる」の意味

```
subtitle-editor/
├── start.bat          ← ダブルクリックするだけ
└── index.html         ← 字幕エディタ本体（これ1ファイル）
```

`start.bat` の中身はたった1行：
```bat
start index.html
```

**Pythonも、Node.jsも、サーバーも不要。**
非エンジニアのオペレーターさんに渡すとき、「このフォルダを渡してstart.batを押してください」
で完結します。

## 「htmxとの相性が悪い」とはどういう意味か

htmxは「ボタンを押す→サーバーが処理→HTMLの一部を差し替える」という仕組みです。

```
[ユーザーが字幕を編集]
  → サーバーにリクエスト送信（ネットワーク通信）
  → サーバーでCPS計算
  → 計算結果のHTMLを返す
  → 画面の一部を差し替え
```

このモデルの問題点：
1. **ネットワーク遅延がある**：CPS計算ごとにサーバーと通信するのは遅すぎる
2. **動画の再生位置との同期が難しい**：動画プレーヤーはブラウザ内でリアルタイム動作するため、サーバーを介せない
3. **サーバーが必要**：singlefileで1ファイル完結という目標と矛盾する

字幕エディタのような「常にリアルタイムで画面が変わるツール」には、
**クライアント（ブラウザ）側だけで完結するReact**の方が圧倒的に適しています。

---

## まとめ：選定の一言理由

| 技術 | 一言理由 |
|------|---------|
| **React 19** | リアルタイムインタラクション（動画同期・CPS計算）はクライアントサイドで完結させる必要があるため |
| **TypeScript** | 複雑な字幕データ構造をミスなく扱うための型定義、AIコード生成の品質向上 |
| **Tailwind v4** | 小規模UIの素早いスタイリング、ファイルを行き来しないシンプルさ |
| **Vite** | 開発速度（即時反映）とビルドの信頼性 |
| **vite-plugin-singlefile** | 非エンジニアへの配布を `start.bat` 1つで完結させるため |

---

## 参考：開発の流れ（イメージ）

```
1. コードを書く（TypeScript + React + Tailwind）
       ↓
2. 開発中は Vite が即座にブラウザに反映
       ↓
3. 完成したら「ビルド」を実行
       ↓
4. vite-plugin-singlefile が1つの index.html に全部まとめる
       ↓
5. index.html + start.bat を zip で渡すだけで配布完了
```
