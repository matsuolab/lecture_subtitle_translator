## 設計の基本方針

### なぜこの設計か

現在の3画面作業（英語テキスト・日本語テキスト・実際の字幕映像）の解消が最終スコープに含まれる。これは「フルオートのCLIパイプライン」ではなく、**人間の介入ポイントを持つパイプライン＋UIが乗る設計** が正しいということを意味する。

- 各ステップを独立モジュールにする → UIは後付けでも繋ぎやすい
- ステップ間のデータを構造化する → どのステップの結果も人間が確認・修正できる
- CLIとして動けば最初は十分 → UIはフェーズ2以降

    ▼
┌─────────────────────────────────┐
│ Step 3: 英訳                     │  ← LLM (改良Gemsプロンプト)

    │
    ▼

## ステップ間のデータ構造（案）

各ステップは以下の構造化データをファイルまたはメモリで受け渡す。

### セグメント（Step 1 出力）

```json
{
  "segment_id": 1,
  "start": 150.0,
  "end": 170.0,
  "ja_raw": "今のAIは空白を検知することができます...",
  "words": [
    {"word": "今の", "start": 150.0, "end": 150.4},
    {"word": "AIは", "start": 150.4, "end": 150.9}
  ]
}
```

### 翻訳済みセグメント（Step 3 出力）

```json
{
  "segment_id": 1,
  "start": 150.0,
  "end": 170.0,
  "ja_clean": "現在のAIは空白を検知することができます。",
  "en_raw": "Current AI can detect silence.",
  "human_review_flag": false
}
```

### Step 4 実装方針（deep-research-report3 より）

**LLMに文字数を数えさせない**。これが最重要原則。

```
推奨パターン：Length Guard（決定論） + LLM Rewriter（確率的）の二層化

① LLM → チャンク分割案を生成
② Python → grapheme cluster数 + CPS を計測（single source of truth）
③ 違反あり → 超過量を構造化してLLMにフィードバック
   例: "Block 3: 52 chars (over by 12), CPS=18.4 (over by 3.4). Shorten while keeping meaning."
④ LLM → 修正案を生成（数えさせない、言い換えだけさせる）
⑤ ②に戻る。N回失敗 → human_review_flag = true
```

**文字数の計測単位**：`grapheme cluster`（Unicode UAX #29）を使う。英語字幕であればコードポイント数と概ね一致するが、絵文字・結合文字が混入した場合の安全のため。

**参考実装**: CEDEC資料の"LengthGuardian"コンセプト、AppTekのバリデーションループ（CPS準拠率を12〜23ポイント向上）

  "segment_id": 1,

## PoCで確定した設計方針（2026-03-24）

### Step 1〜2 の実装方針（PoC検証済み）

**WhisperX文字レベルタイムスタンプ**は日本語でも十分な精度（±0.1秒程度）を確認。
Step 1〜2は以下のパイプラインで実装する。

```
WhisperX JSON（文字レベルTS付き）
  ↓ Gemini全文一括補正（gemini-3-flash-preview）
    - 句読点追加 / フィラー削除 / 誤字修正
    - 全文 > 35,000字のときはギャップ境界でチャンク分割（前後1,500字オーバーラップ）
  ↓ diff-based アライメント（difflib.SequenceMatcher）
    - equal/insert/delete/replace でタイムスタンプを引き継ぐ
  ↓ ギャップ検出（0.30秒）+ 句点分割
  → センテンス単位 SRT
```

**実装コード**: `30_poc/poc_text_correction_alignment.py`（本開発で整理・移植）

### 開発フェーズで対応が必要な課題

| 課題 | 概要 | 優先度 |
|------|------|--------|
| 短セグメントのマージ | チャンク境界付近で数文字のセグメントが発生し表示されない | 中 |
| LLM省略・要約の抑制 | フィラー以外の内容が削除されるケースあり。DELETE率の定量チェックが必要 | 高 |
| 書き起こし精度の限界 | WhisperX自体の誤認識はLLM補正だけでは限界あり | 中 |

## コミュニティ調査から導いた実装方針

> 追記: 2026-03-26
> 根拠: `docs/research/subtitle_community_survey_20260326.md`
> 調査先: SubtitleEdit/Aegisub GitHub Issues、Whisper Discussions、Vikiコミュニティ、Amara

### UI機能の優先順位

#### 🔴 高優先（品質に直結・業界標準）

**1. 1行文字数チェック（42文字/行）**

Whisperユーザーの最大の不満。Netflix基準（42文字）、BBC基準（37文字）は業界コンセンサス。
CPS表示は実装済みだが、1行あたりの文字数チェックが未実装。

```
実装方針:
- SubtitleBlock に「最長行文字数」を計算して表示
- 42文字超でオレンジ警告、55文字超でレッド
- 改行コード \n で行分割して評価
```

**2. flagged（要レビュー）状態のUI活用**

`BlockStatus = 'pending' | 'approved' | 'flagged'` は型定義済みで未使用。
Vikiコミュニティで「フラグ機能がないのに承認フローが存在する矛盾」が繰り返し批判されていた。
「承認」と「要レビュー」の2方向フローがあると品質管理が格段に改善する。

```
実装方針:
- ブロックに「要レビュー」ボタンを追加（承認の反対側）
- flagged ブロックはオレンジ枠で視覚的に区別
- QAパネルで flagged ブロックを一覧表示
```

#### 🟡 中優先（ワークフロー改善）

**3. QAパネル**

どの字幕OSSでも「個別ブロックにCPSは表示されるが一覧は見れない」という不満がある。
問題ブロックを一覧化することでレビュー作業の効率が格段に上がる。

```
実装方針（SettingsTab の隣に QAタブとして追加）:
- CPS超過ブロック一覧
- 1行文字数超過ブロック一覧
- 訳語漏れブロック一覧（すでに findMissingTranslations で検出済み）
- flagged ブロック一覧
- クリックで対象ブロックにジャンプ
```

**4. Find & Replace（正規表現対応）**

表記ゆれ修正の定番。Aegisub・Gaupol・SubtitleEditすべてが持つ基本機能。
用語辞書の自動適応（実装済み）と相補的に機能する。

```
実装方針:
- モーダルで Find / Replace テキストボックス
- 正規表現モードのトグル
- 「現在のブロック」「全ブロック」「未承認のみ」の適用範囲選択
- 置換プレビュー（差分ハイライト）
```

**5. 全体時間シフト**

「動画の前に数秒のカウントダウンが入っていた」「最終的な動画とオフセットがあった」など、
全ブロックを一括でN秒前後に動かしたい場面は頻出する。SubtitleEdit の定番機能。

```
実装方針:
- 設定タブまたはヘッダーに「全体時間シフト: ±N.N秒」入力
- 承認済みブロックも含めて全適用（タイムコード補正のため）
- Ctrl+Z でundo可能
```

#### 🟢 低優先（Release 3〜4以降）

**6. コンテキスト付き再翻訳（Release 2 API接続時）**

SubtitleEdit Issue #9394（2025年2月）で多くの関心を集めた要望。
Geminiの大コンテキストウィンドウを活かして、RAGのRetrieval不要で実現できる。

```
翻訳レイヤー設計（詳細は project_matsuo_subtitle.md 参照）:
L1: 全体要約（講義開始時に1回生成、以降常時埋め込み）
L2: 用語辞書（確定済みGlossaryを毎回埋め込む）
L3: 前後ブロック（再翻訳時に前後N件を渡す）
L4: 全文一括翻訳（初回生成、JSON番号付き出力）

UIへの示唆:
- 設定タブに「再翻訳コンテキスト: 前後N件」スライダー
- 「全体要約を生成」ボタン（パイプラインタブに追加）
```

**7. 翻訳メモリ（Release 4）**

承認済み訳の蓄積・再利用。設計済み（SQLite + sqlite-vec + Gemini Embedding）。

### 差別化ポイントの整理

調査を通じて、本プロジェクトが既存OSSに対して優位な点が明確になった：

| 項目 | 既存OSS | 本プロジェクト |
|------|--------|---------------|
| 動画同期 | デスクトップアプリ必須 | ブラウザ完結・ローカル動画直接読み込み |
| 単一ファイル配布 | インストーラが必要 | HTMLファイル1つ・zip配布可能 |
| 用語辞書自動適応 | 実装例なし（Issue段階） | 実装済み（正規表現 + compromise） |
| 訳語漏れ検出 | なし | 実装済み（日本語原文照合） |
| 翻訳メモリ | Aegisub未実装（Issue #116） | Release 4 で計画済み |
| 承認フロー | 単純な承認のみ | flagged（要レビュー）を含む3状態設計 |
| コンテキスト付き翻訳 | SubtitleEdit で要望段階 | L1〜L4 レイヤー設計済み |
